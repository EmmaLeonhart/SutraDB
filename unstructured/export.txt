How normal vector DBs do HNSW search
Yes, standard HNSW algorithm. The entry point to the top layer is just stored as metadata — when you build the index there's a single designated entry node at the highest layer, and every search starts there regardless of the query vector. It's not derived from the query, it's a fixed global entry point set during index construction. From there you do greedy descent: at each layer find the closest neighbor to your query vector, drop down a layer, repeat, until you're at layer 0 where you do the final beam search for your k nearest neighbors.
So yes — one entry point at the top, all searches start there. Which maps cleanly to your RDF model: that node just has no incoming vertical edges, only outgoing ones descending to layer below.

Vectorized execution
Normal query execution is row-at-a-time — for each row, apply the operator, pass it to the next operator, repeat. Vectorized execution processes a batch of rows (a "vector") through each operator at once. The gains come from a few places:

Better CPU cache utilization since you're operating on contiguous memory
SIMD instructions — modern CPUs can apply the same operation to 4/8/16 values simultaneously in a single instruction
Reduced interpreter overhead per tuple

It's the core reason DuckDB is so fast for analytical queries. For graph workloads it's harder to apply because graph traversal is inherently pointer-chasing rather than operating on contiguous arrays, but for the parts of SPARQL execution that look like relational joins it's applicable.

Rust is a good choice for this. The two usages are totally unrelated:

Vector DB / HNSW → vector as in a point in high-dimensional space, mathematical object

A concrete example: if you're comparing a column of integers against a filter value, scalar execution checks one integer at a time. Vectorized execution loads 8 integers into a single wide CPU register and checks all 8 against the filter in one instruction. Same result, 8x throughput.
So for SutraDB it's only relevant in the "some SPARQL operations look like relational joins over flat data" sense — if you're ever scanning/filtering triple patterns that can be laid out contiguously in memory, you could apply SIMD there. It has nothing to do with your HNSW vectors.
