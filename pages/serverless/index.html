Serverless vs Server Mode

Serverless doesn't mean less. It means you can see where your data lives.

The Misconception

When people hear "serverless database," they often think it's a stripped-down version of a "real" database. That's backwards. A serverless database like SQLite or SutraDB does everything a server-based database does — it just doesn't hide your data behind a daemon process.

Server-based (PostgreSQL, Fuseki)

  • Data stored in an opaque directory managed by a background service
  • You interact through a network port
  • Must install, configure, start the service before doing anything
  • Data location: somewhere in /var/lib/postgres/ or a service directory you never look at
  • Backup: use the server's dump tool

Serverless (SQLite, SutraDB)

  • Data stored in a file you can see: my_graph.sdb
  • You interact by opening the file directly
  • Download the binary, point it at a file, done
  • Data location: wherever you put it
  • Backup: copy the file
Same capabilities, different access model A serverless database is not "less" than a server-based one. It has the same indexes, the same query engine, the same ACID guarantees. The difference is purely about how you access it — directly (file) or indirectly (network port).

Why Serverless by Default?

Instant setup

The SQLite experience: download one binary, run one command, you have a working database. No service installation, no configuration files, no port conflicts, no permission issues.

# That's it. You now have a graph database with vector indexing.
sutra query "SELECT * WHERE { ?s ?p ?o }" --data my_graph.sdb

AI-agent friendly

A key motivation: an AI agent (like Claude Code, Cursor, or a custom tool) can install SutraDB and initialize a database in seconds. No "start the service, wait for it to boot, configure authentication, create a database" ceremony. Just:

  1. Download the binary
  2. Create a .sdb file
  3. Start querying

This matters because AI agents work best when setup is fast and deterministic. A server-based database requires the agent to manage a background process, handle port allocation, deal with authentication — all failure modes that have nothing to do with the actual data work.

Transparency

With a serverless database, you always know exactly where your data is. It's a file in a directory you chose. You can:

  • See it in your file explorer
  • Copy it to another machine
  • Back it up by copying the file
  • Version control it (for small databases)
  • Delete it when you're done
  • Share it by sending the file

With a server-based database, your data is in some system directory managed by the service. You need the server's tools to export, backup, or move it. The data is there, but it's hidden from you.

The Tradeoff: Visibility Cuts Both Ways

The main argument against serverless is exactly the same as the main argument for it: your data is a visible file.

The accidental deletion problem Because a .sdb file sits in a regular directory, it's easy to accidentally delete it, move it, or overwrite it. A server-based database hides data in a system directory that users rarely touch, which provides accidental-deletion protection by obscurity.

This is a real tradeoff, and it's the reason server-based databases exist. But we believe the transparency benefits outweigh the risks, especially because:

  • Users who work with databases know what they're doing. They understand that deleting a data file deletes the data.
  • Explicit is better than implicit. Knowing where your data is and having the power to manage it directly is better than having it hidden "for your own protection."
  • Server mode exists when you need it. If you want the protection of a managed service, start SutraDB in server mode. The same .sdb file works either way.

Server Mode: When You Need It

Serverless is the default, but SutraDB fully supports server mode for cases where you need it:

Use CaseMode
Personal project, scripts, AI agentsServerless (direct file access)
Web application backendServer (HTTP SPARQL endpoint)
Multiple processes reading the same dataServer (shared access via HTTP)
Remote access from other machinesServer (network endpoint)
Jupyter notebook explorationEither (serverless for local, server for shared)
Production deploymentServer (with Docker)
# Serverless: query a file directly
sutra query "SELECT ?s WHERE { ?s a :Person }" --data people.sdb

# Server: start an HTTP endpoint, same file
sutra serve --port 3030 --data people.sdb
# Now accessible at http://localhost:3030/sparql

Same data, same file, same format. The only difference is whether you access it through a file path or a network port.
