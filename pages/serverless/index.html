Deployment: Embedded & Server

Two ways to run the same database. Pick what fits your use case.

Two Modes, One Database

SutraDB can run as an embedded database (open a .sdb file directly, like SQLite) or as a server (HTTP SPARQL endpoint). Both use the same storage format, the same query engine, and the same indexes. The difference is how you access the data.

Embedded (Serverless)

  • Open a .sdb file directly — no daemon, no config
  • Zero setup: download the binary, point at a file, query
  • Data is a file you can see, copy, and move
  • Single-process access only
  • No network overhead

Server Mode

  • HTTP SPARQL endpoint at localhost:3030/sparql
  • Multiple clients can query concurrently
  • Remote access from other machines
  • Optional auth, rate limiting, query timeouts
  • Periodic backups, health monitoring
# Embedded: query a file directly
sutra query "SELECT ?s WHERE { ?s a :Person }" --data people.sdb

# Server: start an HTTP endpoint on the same file
sutra serve --port 3030 --data people.sdb
# Now accessible at http://localhost:3030/sparql

When to Use Each Mode

AI agents (Claude Code, Cursor, etc.)EmbeddedNo process management, no port conflicts, deterministic

Multiple processes reading the same dataServerEmbedded mode is single-process; server handles concurrency
Remote access from other machinesServerEmbedded mode has no network interface
Production deploymentServerAuth, rate limiting, monitoring, backups

Benefits of Embedded Mode

Embedded mode follows the SQLite philosophy: download one binary, run one command, you have a working database. This is the default because it removes every obstacle between you and your data.

  • Instant setup. No service installation, configuration files, port conflicts, or permission issues.
  • Transparent data location. The .sdb file is wherever you put it. Copy it, back it up, share it, version-control it.
  • Agent-friendly. AI agents can install and initialize a database in seconds without managing background processes.
  • No network overhead. Queries go directly to the storage engine, no serialization/deserialization over HTTP.

Benefits of Server Mode

Server mode adds capabilities that embedded mode cannot provide. For anything beyond single-user local access, server mode is the right choice.

  • Concurrent access. Multiple clients, multiple processes, multiple machines — all querying the same database simultaneously.
  • Authentication & access control. Simple passcode auth in the open-source tier; RBAC in the premium tier.
  • Operational safety. Rate limiting prevents runaway queries. Query timeouts kill expensive queries. Periodic backups run automatically.
  • Standard protocol. The SPARQL HTTP Protocol means any SPARQL client (Yasgui, curl, Python's SPARQLWrapper) works out of the box.
  • Health monitoring. The /health endpoint exposes index health, tombstone ratios, and pseudo-table coverage for observability.

Same file, either mode The .sdb file format is the same regardless of how you access it. You can prototype in embedded mode, then deploy the same file in server mode without any migration or conversion.
